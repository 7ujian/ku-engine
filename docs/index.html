Overview

ku is a CLI-based 2D game engine designed for AI agents. AI agents act as both developers (creating scenes, editing nodes) and players (sending input, reacting to state).

Tech Stack

  • Node.js 20+, TypeScript ES2022
  • commander — CLI framework
  • ws — WebSocket server/client
  • matter-js — 2D physics
  • @kmamal/sdl — SDL2 rendering + audio
  • @napi-rs/canvas — CPU canvas rendering
  • vitest — testing

Quick Commands

npm run build          # compile TypeScript
npm test               # run tests
npx ku edit [scene]    # start editor
npx ku play            # spawn play instance
npx ku attach play     # switch to play

Architecture

Dual-instance model: Editor and Play are separate OS processes.

┌──────────┐ WebSocket ┌──────────────────┐ │ ku CLI │ ◄───────────────► │ edit :21200 │ │ │ │ (scene editing) │ │ AI SDK │ ◄───────────────► │ play1 :OS-port │ │ │ WebSocket │ (game loop) │ │ │ ◄───────────────► │ play2 :OS-port │ └──────────┘ └──────────────────┘
ComponentFilePurpose
Editoreditor-runtime.tsPersistent, scene editing, no game loop
Playplay-runtime.tsEphemeral, full game loop, physics + scripts
Instanceinstance.tsBase class: SceneTree + WebSocket + message handler

Instance Lifecycle

ku edit [scene] │ ▼ ┌─────────────────────┐ │ Editor running │◄─────────────────┐ │ (port 21200) │ │ └─────────┬───────────┘ │ │ ku play (snapshot) │ ▼ │ ┌─────────────────────┐ │ │ Play running │ ku stop play │ │ (port 21201) │──────────────────┘ └─────────┬───────────┘ │ ku stop edit │ ▼ Editor stopped

Discovery Files

.ku.edit.pid     → editor process ID
.ku.edit.port    → editor WebSocket port (21200)
.ku.play.pid     → play process ID
.ku.play.port    → play WebSocket port (21201)

Node System

The scene is a tree of typed nodes (Godot-inspired). Nodes are addressed by slash-separated path (e.g. player/sprite).

Transform System

Nodes store local coordinates. World transform computed by composing parent chain:

world = parent_world × local
worldToLocal(node, worldX, worldY)  // physics sync
localToWorld(node, localX, localY)  // render coords

SceneTree.move() preserves world position when reparenting.

Scene Instancing

{
  "id": "player",
  "type": "Node2D",
  "instance": "scenes/player.json"
}

Referenced scene loaded at runtime, merged: instance props override template defaults, children/scripts concatenate. Circular refs detected and rejected.

Expressions

Templates wrapped in {{ }} use recursive descent parser with operator precedence and cross-node references:

{{speed}}                → property reference
{{velocity.x}}           → nested property reference
{{-speed}}               → negated
{{x + 10}}               → arithmetic (+, -, *, /, %)
{{/player/score}}         → cross-node reference
{{random(0, 100)}}        → random float
{{min(3, 7)}}             → minimum
{{max(3, 7)}}             → maximum
{{abs(-5)}}               → absolute value
{{floor(3.7)}}            → floor
{{ceil(3.2)}}             → ceiling

Conditions

{
  "event": "on_frame",
  "condition": {
    "speed": {"gt": 0},
    "/player/dead": {"neq": true}
  },
  "actions": [
    {"set": "score", "to": "{{score + 1}}"}
  ]
}

Operators: eq, neq, gt, lt, gte, lte, in, between

JavaScript Scripts

Nodes can reference a .js file via js_script field. JS scripts run in sandboxed vm context.

// scripts/player.js
let vx = 0;
handlers.on_frame = function(ctx) {
  if (ctx.node.get('moving_right')) vx = ctx.node.get('speed');
  else if (ctx.node.get('moving_left')) vx = -ctx.node.get('speed');
  else vx = 0;
  ctx.node.set('velocity.x', vx);
};
handlers.on_collision = function(ctx) {
  if (ctx.data.other === 'coin') {
    ctx.scene.destroy(ctx.data.other);
    ctx.emit('coin_collected');
  }
};

ctx API: ctx.node.get(prop), ctx.node.set(prop, val), ctx.node.id, ctx.scene.get(path, prop), ctx.scene.set(path, prop, val), ctx.scene.spawn(type, id, props?), ctx.scene.destroy(path), ctx.scene.find(path), ctx.emit(event, data?), ctx.data, ctx.log(...)

Physics

Powered by matter-js. CollisionShape nodes that are children of RigidBody nodes automatically track parent's position/rotation each frame.

RigidBody Properties

PropertyTypeDescription
massnumberBody mass
velocity{x, y}Current velocity
gravity_scalenumberGravity multiplier (0 = no gravity)
linear_dampingnumberVelocity damping
width, heightnumberCollision bounds
angular_velocitynumberRotational velocity

Collision Handling

on_collision fires when RigidBody collides with another RigidBody or Block. on_area_enter fires when any body enters an Area node's zone.

CollisionShape Shapes

  • rect — axis-aligned rectangle
  • circle — circle (use radius)
  • polygon — arbitrary polygon (use points)

Renderer

SDL2 via native bindings. Two-pass rendering: draw + debug overlay.

Sprite Atlas

Sprites can reference texture atlases — single image with named regions in companion JSON:

{
  "texture": "assets/player_sheet.png",
  "regions": [
    {"name": "idle_0", "x": 0, "y": 0, "width": 32, "height": 48},
    {"name": "walk_0", "x": 32, "y": 0, "width": 32, "height": 48}
  ]
}

Sprite rendering modes:

  1. Standalone texture: texture set, atlas empty
  2. Atlas region: atlas + region set → draw named region
  3. Atlas animation: atlas + animation set → play named animation

Coordinate System

  • Origin (0,0) at top-left
  • X increases right
  • Y increases down
  • Units are pixels

Camera2D

Applies viewport offset and zoom to all draws. Supports smoothing for follow-camera behavior.

Client-Server Protocol

WebSocket communication using JSON messages.

Message Format

{
  "type": "command|query|event|response|diff",
  "id": "unique-message-id",
  "instance": "edit|play",
  "payload": { ... }
}

Command → Response

// CLI → Server
{"type": "command", "id": "cmd-001", "payload": {"action": "node.set", "path": "player", "property": "x", "value": 150}}

// Server → CLI
{"type": "response", "id": "cmd-001", "payload": {"ok": true, "data": {"x": 150, "y": 300}}}

State Diff (Server → subscribed clients)

{
  "type": "diff",
  "payload": {
    "frame": 1234,
    "changes": [
      {"path": "player.x", "old": 100, "new": 102}
    ]
  }
}

PluginHost API

  • registerNodeType(type, factory) — register custom node types
  • registerAction(name, handler) — register custom script actions
  • registerMessageHandler(pattern, handler) — register message handlers
  • registerCliCommand(command, handler) — register CLI commands
  • registerNodeRenderer(type, renderer) — register custom node renderers

AI Agent Workflow

The dual-instance model enables a tight edit-test loop for AI agents:

Edit-Test Loop

ku edit # 1. Start editor # ... create/modify scene ... ku play # 2. Spawn play from snapshot ku attach play ku query player.x # 3. Inspect play state ku input key right down # 4. Test interaction ku query diff # 5. Verify behavior ku stop play # 6. Discard play, return to editor ku attach edit # ... fix issues found during testing ... ku play # 7. Re-test with updated scene

Key Design Principles

  • JSON everywhere — all input/output is structured JSON
  • Queryable state — ku query gives full access to game state
  • Diffs — ku query diff shows what changed
  • Deterministic — same inputs produce same outputs
  • Idempotent edits — ku node set player.x 100 always produces same result
  • Composable — commands can be scripted and chained
